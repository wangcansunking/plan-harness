# plan-full

Orchestrate the complete planning workflow end-to-end. This skill runs all plan generation skills in sequence, with user review checkpoints between each phase. It produces a full suite of interconnected planning documents and a dashboard linking them all together.

## When to Use

- When the user wants to generate ALL planning documents for a feature in one session
- When the user says "full plan", "plan everything", "complete planning", "plan-full", or "generate all plans"
- When starting a brand-new feature that needs the complete planning treatment

## What It Produces

All of the following files in `{scenario-path}/`:

| File | Generated By | Description |
|------|-------------|-------------|
| `manifest.json` | `/plan-init` + updates | Scenario manifest with all file references |
| `analysis.html` | `/plan-gen analysis` | Deep codebase analysis — architecture, patterns, conventions, health |
| `design.html` | `/plan-gen design` | Comprehensive design document (reads analysis if present) |
| `state-machine.html` | `/plan-gen state-machine` | Entity state diagrams and transitions |
| `test-plan.html` | `/plan-gen test-plan` | E2E test scenarios with progress tracking |
| `test-cases.html` | `/plan-gen test-cases` | Detailed test cases with interactive harness |
| `implementation-plan.html` | `/plan-gen implementation` | File-level implementation steps |
| `test-report.html` | `/plan-gen test-report` | Live E2E verification report (runs after implementation exists) |
| `dashboard.html` | This skill | Index page linking all documents |

## Prerequisites

- A repository to plan for (auto-detected or user-specified)
- The plan-harness MCP server must be running

## Workflow

### Phase 0: Initialize Context

**Check if `manifest.json` already exists for the target scenario.**

- If YES: Read it and display a summary. Ask the user if they want to re-initialize or continue with existing context.
- If NO: Run the `/plan-init` workflow.

To run `/plan-init`:

1. Detect the repository from cwd or ask the user
2. Call the `plan_get_context` MCP tool to analyze the codebase
3. Call the `plan_list_scenarios` MCP tool to list existing scenarios
4. Let the user select or create a scenario
5. Write `manifest.json`

After initialization, display the context summary and confirm:
> "Context initialized for scenario '{scenarioName}' in {repoName}. Ready to begin planning?"

### Phase 1: Gather Feature Description

Ask the user to describe what they want to build. This description drives all downstream planning.

Prompt:
> "Describe the feature or change you want to plan. Include:
> - What does it do? (core functionality)
> - Who is it for? (target users)
> - What systems does it touch? (frontend, backend, database, services)
> - Any constraints or requirements? (timeline, compatibility, security)
> 
> Be as detailed as possible -- this drives all the planning documents."

Store the full description. If the user's description is very brief (< 50 words), ask follow-up questions:
- "Can you elaborate on the user interaction? What does the UI look like?"
- "What data needs to be stored or processed?"
- "Are there any external services or APIs involved?"

Update `manifest.json` with the enriched description.

### Phase 2: Generate Analysis Document

Run the `/plan-gen analysis` workflow. **Analysis is the problem statement + code-logic reading** for the whole run — current state (product flow AND the code paths this plan will touch), what we're solving, pain points (business + code-level, each cited by file/line), root causes, impact. No solutions (those belong in design).

1. Read `manifest.json` for repo path + scenario context + the user's Phase 1 feature description
2. Dispatch the PM agent (frame: problem to solve / business-level pain points / impact / urgency / constraints, using the Phase 1 description as the brief)
3. Dispatch the Architect agent (ground: read the actual relevant modules — control flow, data flow, surface bugs / anti-patterns / silent failures / N+1 / dead paths; cite file+line for every finding; map each finding to a root cause layer: logic / abstraction / architecture / external / historical)
4. Dispatch the Writer agent (assemble sections 1–6 per `skills/plan-gen/types/analysis.md`; §1 and §3 must carry file+line citations)
5. Write `{scenarioPath}/analysis.html`
6. Update manifest.json (`analysisHtml`, `analysisGeneratedAt`)

**USER REVIEW CHECKPOINT:**

```
=== Analysis Document Review ===

Problem to solve:
  {one-line summary from §2}

Observed pain points:
  P1 {…}   P2 {…}   P3 {…}   …

Root causes:
  {grouped bullets}

Impact:
  {who / how often / severity}

Questions for you:
- Is the problem framed right? Anything missing from the pain points?
- Do the root causes actually explain the pain points, or are we treating symptoms?

Type your feedback, or press Enter to continue to Phase 3 (Design).
```

If the user provides feedback, re-dispatch PM + Architect with the correction, regenerate analysis.html, and re-show the summary. Treat analysis as a decision doc — design hangs off it.

If the user passed `--skip-analysis` or re-running with an existing analysis.html that's still accurate, skip this phase.

### Phase 3: Generate Design Document

Run the `/plan-gen design` workflow:

1. Read `manifest.json` for context; also read `analysis.html` if present (optional input)
2. Dispatch the Architect agent (system design, data models, API contracts, architecture diagrams) — grounded in the analysis findings
3. Dispatch the PM agent (requirements, acceptance criteria, use cases, scope)
4. Dispatch the Writer agent (combine into design.html)
5. Write `{scenarioPath}/design.html`
6. Update manifest.json

**USER REVIEW CHECKPOINT:**

After generating the design, present key decisions for review:

```
=== Design Document Review ===

Key architectural decisions:
1. {Decision 1 summary}
2. {Decision 2 summary}
3. {Decision 3 summary}

Data model entities: {list}
API endpoints: {count} endpoints across {n} controllers
Use cases: {count} use cases with {n} acceptance criteria

Questions for you:
- Does the data model look right? Any entities missing?
- Are the API endpoints what you expected?
- Any use cases that should be added or removed?

Type your feedback, or press Enter to continue to the next phase.
```

**If the user provides feedback:**
1. Re-dispatch the relevant agents with the feedback incorporated
2. Regenerate design.html
3. Show the updated decisions
4. Repeat until the user approves

**If the user presses Enter (no feedback):**
Proceed to Phase 4.

### Phase 4: Generate State Machine Document

Run the `/plan-gen state-machine` workflow:

1. Read `manifest.json` and `design.html`
2. Dispatch the Architect agent (extract entities, states, transitions)
3. Dispatch the Writer agent (format into state-machine.html with SVG diagrams)
4. Write `{scenarioPath}/state-machine.html`
5. Update manifest.json

Brief confirmation (no full review checkpoint here):
```
State machine document generated with {n} entities and {n} state diagrams.
```

### Phase 5: Generate E2E Test Plan

Run the `/plan-gen test-plan` workflow:

1. Read `manifest.json` and `design.html`
2. Dispatch the PM agent (extract testable requirements)
3. Dispatch the Tester agent (design E2E scenarios)
4. Dispatch the Writer agent (format into test-plan.html)
5. Write `{scenarioPath}/test-plan.html`
6. Update manifest.json

Brief confirmation:
```
E2E test plan generated with {n} scenarios (P0: {n}, P1: {n}, P2: {n}).
```

### Phase 6: Generate Test Cases

Run the `/plan-gen test-cases` workflow:

1. Read `manifest.json`, `design.html`, and `test-plan.html`
2. Dispatch the Tester agent (generate detailed test cases)
3. Dispatch the Frontend Dev agent (design interactive harness UI)
4. Dispatch the Writer agent (combine into test-cases.html)
5. Write `{scenarioPath}/test-cases.html`
6. Update manifest.json

Brief confirmation:
```
Test cases generated: {n} total (P0: {n}, P1: {n}, P2: {n}) across {n} categories.
```

### Phase 7: Generate Implementation Plan

Run the `/plan-gen implementation` workflow:

1. Read `manifest.json` and `design.html` (plus state-machine.html and test-plan.html if available)
2. Dispatch ALL six agents:
   - Architect: implementation sequence, dependency graph
   - PM: milestones, acceptance checkpoints
   - Frontend Dev: frontend implementation steps
   - Backend Dev: backend implementation steps
   - Tester: per-step test requirements
   - Writer: combine into implementation-plan.html
3. Write `{scenarioPath}/implementation-plan.html`
4. Update manifest.json

Brief confirmation:
```
Implementation plan generated: {n} steps across {n} phases.
```

### Phase 8: Generate Dashboard

Generate a `dashboard.html` file that serves as the index/landing page for all plan documents.

Write the dashboard directly (no agent dispatch needed for this):

```html
The dashboard.html should contain:

<nav> — Same navigation bar as other plan files, with "Dashboard" highlighted

<header> — Dashboard header:
  - Scenario name and description
  - Generation date
  - Repo and tech stack info

<main> — Document cards grid:
  For each generated document, a card with:
  - Document name (linked to the file)
  - Icon or visual indicator of document type
  - Description of what it contains
  - Key stats (e.g., "15 API endpoints", "32 test cases", "18 implementation steps")
  - Last generated timestamp
  - Status badge (Generated / Not Yet Generated)

  Cards for (in canonical workflow order):
  1. Analysis (analysis.html)
  2. Design Document (design.html)
  3. State Machine (state-machine.html)
  4. E2E Test Plan (test-plan.html)
  5. Test Cases (test-cases.html)
  6. Implementation Plan (implementation-plan.html)
  7. Test Report (test-report.html, if generated)

<footer> — Metadata:
  - Scenario: {name}
  - Repository: {repoName}
  - Generated by: plan-harness
  - Date: {timestamp}
```

Styling requirements for dashboard.html:
- Self-contained: ALL CSS inline in a `<style>` tag
- Card grid: 2-3 columns, responsive
- Cards: hover effect with shadow, large click area
- Color scheme consistent with other plan files (#0078D4 primary)
- Status badges: green for generated, gray for not yet generated
- Typography: system font stack
- Navigation bar matching sibling files

Write `{scenarioPath}/dashboard.html`.

### Phase 9: Serve and Summarize

1. Call the `plan_serve_dashboard` MCP tool to open the dashboard in a browser (if available).
2. If the MCP tool is not available, tell the user the file path to open manually.

Display a final summary:

```
=== Full Planning Complete ===

Scenario:   {scenarioName}
Repository: {repoName}
Path:       {scenarioPath}

Generated Documents:
  [Y] analysis.html            — Codebase analysis: architecture, patterns, health
  [Y] design.html              — Design document with architecture, API, data model
  [Y] state-machine.html       — Entity state diagrams and transitions
  [Y] test-plan.html           — E2E test plan with {n} scenarios
  [Y] test-cases.html          — {n} test cases with interactive harness
  [Y] implementation-plan.html — {n} steps across {n} phases
  [Y] dashboard.html           — Index page linking all documents

Open dashboard.html in your browser to navigate between documents.

Each HTML file is self-contained and can be:
  - Opened directly in any browser
  - Shared with teammates (no server needed)
  - Printed as PDF for offline review
  - Used as a living checklist during implementation

To re-generate any individual document, run the corresponding skill:
  /plan-gen analysis, /plan-gen design, /plan-gen state-machine, /plan-gen test-plan, /plan-gen test-cases, /plan-gen implementation, /plan-gen test-report
```

## Iterative Feedback Loop

At every USER REVIEW CHECKPOINT, the workflow supports iteration:

1. User provides feedback (text description of changes wanted)
2. Identify which agents need to re-run:
   - Architecture changes -> re-dispatch Architect, then cascade: Writer for design.html, then downstream docs
   - Requirements changes -> re-dispatch PM, then Writer
   - UI changes -> re-dispatch Frontend Dev
   - Test changes -> re-dispatch Tester
3. Regenerate the affected HTML file
4. Show updated summary
5. Ask for approval again

If the user says "redo everything" or "start over", re-run from Phase 2 with the updated description.

## Error Handling

| Error | Resolution |
|-------|------------|
| MCP server not running | Warn user; attempt manual file operations (create dirs, write files directly) |
| Any individual phase fails | Log the error, ask user if they want to retry or skip that phase |
| User abandons mid-workflow | All previously generated files remain valid and usable |
| Design feedback loop exceeds 3 iterations | Suggest: "Consider finalizing the design and iterating during implementation instead." |
| Disk space or write permissions | Report the specific error; suggest an alternative output directory |
| Agent dispatch fails | Retry once; if still failing, generate a simplified version inline |
| Previous documents exist | Ask at the start: "Existing plan documents found. Start fresh (overwrite all) or continue from where you left off?" |
| Prompt template file not found | Use the inline prompts from the individual skill definitions |
